run-validation Guide#

Overview#

run-validation is the master orchestrator. It reads a single master_config.yaml and runs all configured analysis steps in sequence, handling logging, error recovery, and simulation registry updates.

Use it when you want to run all validation types for an experiment in one command. For individual analysis types, run the dedicated scripts directly (e.g. argo-profiles, tidal-analysis).

Config structure#

Individual per-analysis configs are in config/: config/gridded_2d_validation.yaml · config/tidal_analysis.yaml · config/argo_profiles.yaml

# master_config.yaml

experiment:
  area:          NS
  experiment_id: Baseline
  name:          "Baseline run 2015–2022"

model:
  base_path:        /data/model/NS/
  filename_pattern: "ns_{year}.nc"
  name:    pyGETM
  version: ""

output:
  base_dir: ./analyses

workflow:
  years: [2015, 2022]

analyses:
  gridded_2d:
    enabled: true
    config: config/gridded_2d_validation.yaml

  gridded_3d:
    enabled: false
    config: config/gridded_3d_validation.yaml

  tidal_analysis:
    enabled: true
    config: config/tidal_analysis.yaml

  argo_profiles:
    enabled: true
    config: config/argo_profiles.yaml

  cruise_ctd_profiles:
    enabled: false
    config: config/cruise_ctd_profiles.yaml

Required top-level sections: experiment, analyses, output, workflow.

Usage#

# Run all enabled steps
run-validation --config master_config.yaml

# Run a single step (useful for debugging or re-running one analysis)
run-validation --config master_config.yaml --step tidal_analysis

# List all recognised step names
run-validation --list-steps

Error handling#

Each step runs independently. If one step fails it is logged and the orchestrator continues with the remaining steps. Failed steps are listed in the final summary. Exit code is non-zero if any step failed.

Partial runs with –step#

--step is useful when:

  • A single step failed and needs to be re-run after fixing the config.
  • You want to add a new analysis type without re-running everything.
  • You are debugging a specific script.
# Re-run only tidal analysis after fixing the tidal config
run-validation --config master_config.yaml --step tidal_analysis

Relationship to individual scripts#

run-validation delegates to the same individual scripts that you can call directly. The per-step config files (e.g. config/argo_profiles.yaml) are passed through unchanged. The master config adds the experiment block that the individual scripts normally get from .stats.yaml or command-line flags.

For fine-grained control — custom flags, --no-model, --save-residuals — run the individual scripts directly instead of going through the orchestrator.
