A systematic platform for validating regional ocean model output against observations and for evaluating future climate scenarios. The system is deliberately model-agnostic: any model that writes NetCDF or Zarr output can be connected by describing its coordinate naming conventions in a single YAML file — no source-code changes required.

Workflow#

data-loader ──► xarray Dataset ──► validation / scenario tools
                                          │
                                    plots + tables
                                          │
                              ┌───────────┴───────────┐
                           PDF report            Hugo website

Step by step#

  1. Load — data-loader opens model output and observation files from local disk, Copernicus Marine (CMEMS), CMIP6 (PANGEO catalog), Argo, ICES/EMODnet Chemistry, or GLODAP. All sources are accessible through a unified API — as a CLI command, a YAML configuration block, or a Python call (DataLoader.load_from_config()).

  2. Validate / analyse — the appropriate analysis tool compares model and observations using RMSE, bias, MAE, and correlation. Multiple analysis types are supported (see below). A master config (run-validation) can chain all steps in one command.

  3. Bias-correct — the bc command applies quantile delta mapping (QDM) to CMIP6 atmospheric forcing fields for use in future-scenario ocean runs.

  4. Report — ocean-reporting walks the analyses/ tree and produces either a multi-chapter PDF or a full Hugo website with interactive tables, sortable simulation lists, and linked plots.

Areas and experiments#

Every simulation is identified by exactly two labels:

  • Area — a geographic domain (e.g. NS for the North Sea, ENA8 for the extended North Atlantic at 1/8°). Each area has a documented geographic extent, grid resolution, and depth range.

  • Experiment — a named model configuration within that area (e.g. Baseline, ObsKd, Adaptive_2). Experiment names are free text but must be unique within the area.

Together they uniquely identify a simulation and determine where all outputs are written:

analyses/
└── areas/
    └── <AREA>/
        └── validations/
            └── <EXPERIMENT>/
                ├── plots/
                └── tables/

This layout is enforced by all tools and understood by ocean-reporting. The simulation registry (simulation_list.db) tracks metadata, run timestamps, and reporting status for every (area, experiment) pair.

Model agnosticism#

Ocean models use a wide variety of coordinate naming conventions. This system resolves coordinate names at runtime via config/name_mappings.yaml — a lookup table that maps more than 20 variants (latt, nav_lat, TLAT, latitude, …) to standard names.

Modellatlon
pyGETM (this project)lattlont
NEMOnav_latnav_lon
ROMSlat_rholon_rho
CESMTLATTLONG
ERA5latitudelongitude

Adding a new model requires only adding its coordinate names to the YAML file — no Python changes.

Statistical metrics#

All validation types compute the same core set of metrics:

MetricDescription
RMSERoot mean square error
BiasMean difference (model − observation)
MAEMean absolute error
CorrelationPearson correlation coefficient
NNumber of valid comparison points

Rankings combine RMSE, bias, and correlation using weighted scores to identify the best-performing experiment within each area.

Installation#

git clone https://github.com/bolding/stats.git
cd stats
conda env create -f environment.yml
conda activate stats
pip install -e .

Optional dependencies unlock additional data sources:

pip install -e ".[copernicus]"   # Copernicus Marine / CMEMS
pip install -e ".[cmip6]"        # CMIP6 via PANGEO catalog
pip install -e ".[animations]"   # MP4 / HTML animations

Use of AI in this project#

This project uses Claude Code (Anthropic) as a development assistant. The division of responsibilities is as follows:

Human (Karsten Bolding):

  • Scientific design: choice of metrics, validation strategies, analysis types
  • Domain knowledge: ocean modelling, coordinate conventions, observation datasets
  • Architectural decisions: directory layout, configuration design, tool interfaces
  • Quality control: reviewing all generated code and content before use
  • Writing YAML configurations and running analyses

Claude Code (AI assistant):

  • Implementing Python modules and CLI scripts from specifications
  • Refactoring and extending existing code
  • Writing and updating Hugo content pages (including this page)
  • Suggesting code structure and catching implementation errors
  • Generating documentation drafts for human review

All code and content produced with AI assistance is reviewed and accepted by the human author before being committed. The AI does not have access to model output data or observation files, and does not make scientific judgements about the validity of results.
